Create PROJECT-BLUEPRINT.md at the project root. This file must stay scannable forever, even at 500+ entries — it's for recognition, not reading. No prose, no paragraphs, no explanations. Pure tables and tags.

Design rules (apply to every section):

Every entry is one row/line, max ~8-10 words. If it needs more, it's the wrong place — put a link to the actual file/code instead.
Use fixed short status codes everywhere, never write them out: D=done, P=in-progress, X=planned/pending, R=resolved, !=blocking/critical
Every section starts with a one-line index of tags/IDs in it (so you scan the index line first, jump straight to the matching row, never read the whole table top to bottom)
Newest entries always added at the top of their table, not the bottom — recent = most relevant, no scrolling through old history to find current state

On any task: check section 4 and 5 rows relevant to the task's keywords only — scan the ID/tag index line first, jump to matching rows, ignore the rest.
After finishing a task: add/update rows silently, no announcement, no explanation of what you're logging.
When any table crosses 25 active rows, archive the oldest as described in section 6 — do this automatically without being asked.
Never write a sentence where a row will do. If you're about to write more than one line for an entry, stop and shorten it or split it into Area+ID+Status instead.
